第一章 碰撞系统的三维架构

1.1 物理层面的实体交互

当刻晴的星斗归位撞击丘丘人盾牌时迸发的金属火花，或是诺艾尔挥舞巨剑将敌人砸向岩壁时触发的额外伤害，这些都属于基础物理碰撞的范畴。游戏通过实时计算角色攻击判定的碰撞体积（Hitbox）与敌人的受击体积（Hurtbox）重叠程度，结合速度、角度等参数动态生成击退效果。值得注意的是，地形因素会显著改变碰撞收益——将敌人击飞至水域会触发"落水即死"机制，而利用悬崖边缘的碰撞判定期限能实现地形杀这种高效清场手段。

1.2 元素反应的链式引擎

元素附着与碰撞系统的结合堪称米哈游的神来之笔。当可莉的蹦蹦炸弹（火元素）与莫娜的虚影（水元素）在空气中相撞，触发的"蒸发"反应不仅是伤害数字的简单叠加——碰撞瞬间产生的冲击波会重新计算敌人的受击状态，打断其当前动作并重置元素附着计时器。这种机制使得元素大师们能通过精确的碰撞时序控制，实现"冻结→碎冰→超导"的连锁控制。

1.3 怪物受击的差异化反馈

对比史莱姆被属性克制时的夸张击退与遗迹守卫对轻型攻击的霸体状态，碰撞系统实际上内置了一套复杂的"质量-韧性"计算公式。例如面对愚人众先遣队时，雷电将军的梦想一刀能直接穿透多个敌人的碰撞体积形成串刺效果，而早柚的翻滚攻击则会被大型敌人的质量抵消击退效果——这种差异正是策略搭配的核心依据。

Mihomo 如何启用 TUN 模式？
在 Mihomo 的配置文件中启用 `tun:` 模块并安装系统驱动即可实现全局代理。TUN 模式可将所有系统流量导向代理核心，实现真正意义上的全局加速。
SSR 是否支持 UDP 协议转发？
原版 SSR 并不完全支持 UDP 转发，但部分魔改版本（如 SSRR）支持该功能。可在配置中开启 udp-over-tcp 模式或使用 Socks5 UDP relay 方案实现游戏等实时通信的代理需求。
如何判断节点是否支持 UDP 转发以便游戏加速？
可通过客户端的协议与节点说明确认是否支持 UDP（例如支持 UDP Relay 或 TUN/TAP 模式）。也可实际用 UDP 测试工具或直接用目标游戏连接进行延迟/丢包测试以验证是否可用。
Clash 的自动测速功能如何设置？
Clash 支持定期测速节点延迟，用户可以设置测速频率和超时时间。测速结果会影响策略组的节点选择，从而自动切换至最优节点，提高访问速度和连接稳定性。
如何在 Clash for Android 中导入订阅链接？
Clash for Android 支持通过订阅链接快速导入节点。用户只需在配置界面中点击“新建配置”，选择“URL 导入”并粘贴订阅地址即可，程序会自动下载并解析所有节点配置。
Mihomo 的 Rule-Set 与 Clash Premium 是否兼容？
Mihomo 完全兼容 Clash Premium 的 Rule-Set 格式，支持相同的远程规则订阅与策略分组。用户可直接迁移原 Clash 配置，无需额外修改或转换。
PassWall2 的主要特性有哪些？
PassWall2 是基于 OpenWRT 的代理管理面板，支持多种协议（V2Ray、Trojan、Shadowsocks 等），拥有图形化界面和分流规则设置功能。它能帮助路由器实现全局代理和智能分流。
Mihomo 启动后 CPU 占用高，有哪些优化建议？
检查是否启用了大量日志或复杂脚本，关闭不必要的规则提供者与日志调试；在多核服务器上启用并发或调整线程数；更新到最新版本以获得性能优化，并监控是否有异常连接导致处理压力。

Surfboard 是一款基于 Clash 核心的 iOS 应用，专为苹果用户设计。它通过友好的界面实现分流管理、节点切换和延迟测试等功能。与 Clash 类似，Surfboard 也支持订阅更新和多种代理协议，兼容性非常好。
